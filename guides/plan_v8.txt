
DREDGE v6 → v8 Migration Plan (TDD Style)
Overview
This plan follows strict TDD (Red → Green → Refactor) across 8 major phases, with each phase broken into test-first steps. The plan assumes you're starting from a clean v6.0 baseline and moving through v6.1 stabilisation to Phase 8 (Agentic Domain Discovery).

Phase 0: Pre-Flight Checks & Branch Strategy
Branch Structure
text

├── feat/v6.1-redis-stabilisation
├── feat/phase7-provider-layer
├── feat/phase7b-shadow-adapter
├── feat/phase8-domain-discovery
└── feat/phase8.5-execution-model (parallel track)

DREDGE v6 → v8 Roadmap: Summary Analysis
Phase v6.1 — Redis Stabilisation
What it does
Moves in-memory state (rate limiter token buckets and availability cache) to Redis, adds health checks, and implements graceful fallback to in-memory mode when Redis is unavailable.

Why it's in the roadmap
Required before horizontal scaling. Without this, running multiple orchestrator instances would allow each to have independent rate limit counters, potentially exceeding API limits by 2× or more. Also ensures availability data survives restarts.

Strengths
Zero user-facing changes — purely operational improvement

Graceful degradation — app continues working if Redis is down

Enables horizontal scaling — prerequisite for handling more users

Low risk — existing APIs unchanged, fallback preserves single-instance behaviour

Weaknesses
Adds operational complexity (Redis becomes a dependency)

Fallback mode masks Redis failures in production if not monitored

No immediate user benefit — pure infrastructure work

UX impact
None directly. Indirectly enables future phases that require multi-instance deployments, which improves reliability and scalability for all users.

Phase 7 — Provider Layer & Generic Adapter
What it does
Formalises a Provider interface and implements three new providers (CSV, XLSX, PDF). Introduces GenericAdapter that reads a sources array from DomainConfig and fans out to appropriate providers. Adds per-source refresh policies.

Why it's in the roadmap
Decouples transport logic from domain logic. Previously, adding a new data format required writing a bespoke adapter. Now domains can mix multiple source types (REST API + CSV supplement) with minimal code.

Strengths
Dramatically reduces engineering overhead for new data sources

Enables multi-source domains — combine REST APIs with static files

Clean separation of concerns — providers handle transport, adapters handle merging

Per-source TTL — more granular cache control

Weaknesses
Adds abstraction layer — steeper learning curve for new contributors

Merge logic can become complex with conflicting data from multiple sources

PDF extraction quality varies wildly — may produce unusable data

UX impact
Users get access to more data types without waiting for custom development. A domain that previously only had REST API data might suddenly include historical CSV supplements, enriching results. No direct UI changes, but result quality improves.

Phase 7b — Shadow Adapter Pattern
What it does
Adds an automatic fallback layer that fires when a primary source fails or returns empty. Uses an agent (Mastra + Stagehand) to discover alternative sources, fetch samples, map to existing schema, and register in ApiAvailability. Subsequent queries use the discovered source in parallel with the primary.

Why it's in the roadmap
Most data gaps are coverage gaps within existing domains, not missing domain types. The shadow adapter fills these gaps cheaply and automatically, improving success rates without requiring new domain development.

Strengths
Invisible to users on success — they just get data instead of empty results

Learns over time — discovered sources become co-primary on subsequent queries

No schema changes — maps to existing domain schemas

Low risk — primary path unchanged, shadow only fires on failure

Warm path becomes fast — agent overhead only on first miss

Weaknesses
Agent overhead on first miss — can add seconds to response time

Stagehand dependency — brittle if target websites change structure

LLM mapping accuracy — may incorrectly map columns, producing wrong data

No user visibility — users don't know data came from a shadow source

UX impact
Users see dramatically fewer empty results. A query that previously returned "no data available" now returns actual data, often from alternative sources the user didn't know existed. Response time may be slower on first query (as agent works), but subsequent queries are fast.

Phase 8 — Agentic Domain Discovery
What it does
When a query can't be matched to any registered domain, an agent automatically discovers suitable public data sources, proposes a schema, and registers a new domain — with human approval required before going live.

Why it's in the roadmap
Reduces manual effort of adding new data types to zero for straightforward sources. Enables the platform to grow its domain coverage organically based on actual user queries.

Strengths
Self-growing platform — new domains appear based on user demand

Human-in-the-loop — prevents bad schemas from being registered

Audit trail — all discovery attempts logged

Retries original query — user gets result after discovery completes

Weaknesses
Human approval required — can't go fully autonomous without proving accuracy

Schema proposal accuracy varies — LLMs sometimes hallucinate fields

Migration risk — automated schema generation could produce invalid migrations

Slow path — discovery + approval could take minutes

UX impact
Transformative for power users. A query about a data type DREDGE has never seen before now returns data after a short delay, rather than "domain not found". Over time, the platform becomes more comprehensive without explicit engineering work. Approval UI gives transparency into what's being created.

Phase 8.5 — Query Execution Model
What it does
Introduces immutable QueryRun and DatasetSnapshot entities. Each execution creates a snapshot with checksum. Workspaces reference snapshots, not live queries. Results don't change between sessions without explicit user action.

Why it's in the roadmap
Prevents a well-known analytics platform failure mode: silent data changes under pinned visualisations. Required before collaborative workspaces (Phase 11) to ensure reproducibility.

Strengths
Reproducible results — snapshots can be traced back exactly

Safe background refresh — new snapshots don't break old dashboards

Clear audit trail — know exactly which sources and schema version produced each result

Enables versioned packages — marketplace updates don't break existing workspaces

Weaknesses
Storage growth — snapshots accumulate over time

Conceptual complexity — users must understand snapshot vs live query

No automatic updates — dashboards become stale until manually refreshed

UX impact
Major change for workspace users. Dashboards become stable — a chart pinned on Monday looks the same on Wednesday even if new data arrived. Refresh becomes an explicit action with clear expectations. For casual users running one-off queries, nothing changes. For analysts building reports, this is critical for trust and reproducibility.

Phase 9 — Multi-Source Domain Enrichment
What it does
Allows a single domain to draw from multiple heterogeneous sources simultaneously (REST API + weekly CSV + scraped supplement). Adds deduplication, conflict resolution, source tagging, and background refresh scheduling.

Why it's in the roadmap
Real-world data is rarely in one perfect place. Enrichment combines sources to provide the most complete dataset possible, while the execution model ensures snapshots remain stable.

Strengths
More complete data — combines strengths of multiple sources

Source traceability — know which source contributed each row

Automated refresh — keeps data current without manual intervention

Preserves snapshots — old dashboards unaffected by refreshes

Weaknesses
Merge complexity — deciding which source "wins" on conflicts is non-trivial

Refresh scheduling complexity — coordinating across source types

Storage duplication — each refresh creates new snapshot

UX impact
Users get richer, more complete datasets. A crime query might now include data from police API + court records CSV + news scrapes, all merged seamlessly. The source tag in exports helps advanced users understand data provenance. Dashboard users see notification when newer snapshots are available.

Phase 10 — Semantic Query Layer
What it does
Replaces keyword-based intent detection with embedding-based semantic classification using pgvector. Enables concept-level queries ("deprivation", "liveability") and composite intents that span multiple domains with LLM synthesis.

Why it's in the roadmap
Moves beyond literal matching to understanding what users actually want. Enables queries that don't match domain names exactly and complex questions that require synthesising multiple data types.

Strengths
Understands user intent — not just keyword matching

Concept-level queries — "safety" combines crime + lighting + accident data

Composite results — unified narrative across domains

Improves with usage — more example queries → better matching

Weaknesses
Vector infrastructure — requires pgvector extension, embedding API costs

Synthesis quality varies — LLM narratives can be wrong or misleading

Slower queries — embedding + similarity search + synthesis adds latency

Harder to debug — why did this query match that domain?

UX impact
Transformative. Users can ask natural questions ("is this area safe to live in?") and get thoughtful answers synthesising multiple data sources, not just raw crime stats. The interface becomes conversational. Power users still have access to raw data, but casual users get insights.

Phase 11 — Collaborative Workspaces
What it does
Adds persistent workspaces where teams can save queries, annotate results, and build dashboards. Saved queries reference snapshots, ensuring stability. Exports embed traceability metadata.

Why it's in the roadmap
Moves DREDGE from single-session tool to persistent analytical platform. Enables team collaboration, report generation, and the commercial model (team subscriptions).

Strengths
Persistent state — work doesn't disappear between sessions

Team collaboration — share insights, add annotations

Stable dashboards — pinned results don't change unexpectedly

Auditable exports — every chart traces back to its snapshot

Weaknesses
Auth complexity — user management, permissions, sharing

Storage growth — workspaces, queries, annotations accumulate

UX complexity — more features mean steeper learning curve

Requires Phase 8.5 — can't build without snapshot model

UX impact
Major shift. Users go from "run query, get result, leave" to "build dashboard, share with team, revisit next week". Annotations enable collaboration and institutional memory. PDF exports with embedded metadata satisfy compliance requirements. This is where DREDGE becomes a platform, not just a tool.

Phase 12 — Public Data Marketplace
What it does
Allows domain configurations generated by the agent pipeline to be published, rated, and reused across DREDGE instances. Includes package registry, versioning, signing, and trust scoring.

Why it's in the roadmap
Builds a shared library of community-contributed data sources. Prevents duplicate work — one user's discovered domain becomes available to all. Creates network effects and community around the platform.

Strengths
Network effects — more users → more domains → more value

Reduces duplicate discovery — agent checks registry first

Community validation — ratings and usage indicate quality

Versioned packages — updates don't break existing snapshots

Weaknesses
Security risks — community packages could be malicious

Quality variance — some packages will be poorly configured

Registry operation — requires separate service, moderation

Trust scoring complexity — preventing gaming the system

UX impact
Users gain instant access to domains discovered by others. A query about a niche dataset might succeed immediately because someone else already discovered and published it. Power users can contribute back to the community. Trust scores help users decide which packages to rely on.

Phase 13 — Query Result Abstraction Layer (Deferred)
What it does
Replaces domain-per-table storage with universal QueryResult envelope stored in single table with JSONB columns. All adapters output standardised structure with metrics, attributes, geometry.

Why it's in the roadmap
Simplifies multi-source enrichment, composite intents, marketplace security, and agentic discovery. But it's deferred until the pain of per-domain tables is actually felt (marketplace growth, schema maintenance burden).

Strengths
Unified storage — single table for all results

Simplified queries — no dynamic table names

Marketplace security — no migration SQL in packages, just flattenRow

Generic visualisations — metric + geometry → map, metric + time → chart

Agentic discovery simpler — no model generation, just flattenRow mapping

Weaknesses
Breaking change — requires migrating all existing data

JSONB performance — indexing more complex than columns

Loss of type safety — database no longer enforces per-domain structure

Tooling impact — existing reports expecting specific columns break

UX impact
Invisible to end users if migration is handled correctly. For developers and marketplace contributors, dramatically simplifies package creation — no more writing migrations. Visualisations become more consistent across domains since they're metric-driven rather than domain-specific.

DREDGE: Architectural Assessment & Strategic Analysis
Overall Viability Assessment: 7.5/10
DREDGE is architecturally sound but strategically ambitious. The platform addresses a real problem (unified access to fragmented public data) with a well-thought-out technical foundation. However, several assumptions and choices warrant scrutiny.

Market Viability Factors
Factor	Assessment	Risk Level
Problem relevance	High - public data is genuinely fragmented	Low
Technical feasibility	High - components exist, patterns proven	Low
User acquisition	Medium - requires critical mass of domains	Medium
Commercial model	Medium - workspaces as paid tier is plausible	Medium
Competitive landscape	High - many point solutions exist	High
Data quality liability	High - community sources may be unreliable	High
The core insight is correct: public data is siloed, inconsistent, and hard to combine. The execution risk lies not in the technology but in whether the platform can achieve sufficient domain coverage and data quality to become indispensable.

Architecture Quality Assessment
Strengths ✅
1. Clean Separation of Concerns
text
Query Pipeline → Domain Registry → Adapter → Provider
Each layer has a single responsibility. query.ts remains domain-agnostic, adapters handle domain logic, providers handle transport. This is textbook and will age well.

2. Immutable Execution Model (Phase 8.5)
Adding snapshots before workspaces shows foresight. Most platforms add this after breaking user dashboards and regretting it. The SHA-256 checksum on rows is a nice touch for detecting silent corruption.

3. Graceful Degradation
Redis fallback to in-memory, provider failures not failing entire queries, shadow adapter only on empty results — the system is designed to keep working when parts break. This is production thinking, not prototype thinking.

4. Provider Interface
Abstracting transport (REST/CSV/XLSX/PDF) behind a uniform interface means adding a new format (e.g., Parquet, Shapefile) doesn't require rewriting adapters. This is the kind of extensibility that pays dividends over years.

5. Shadow Adapter Pattern
Elegant solution to the "some data missing" problem. The warm path optimisation (parallel fetch after discovery) shows attention to real-world usage patterns.

Concerns ⚠️
1. Over-Abstraction Risk
text
GenericAdapter → Provider Interface → Provider Implementations → Source Configs → FlattenRow Maps
There are many layers between a user query and actual data. Each layer adds cognitive load for developers and potential failure modes. Phase 13's result abstraction adds another layer. At what point does flexibility become complexity?

2. LLM Dependency Density
The roadmap relies on LLMs for:

Intent parsing (existing)

Schema mapping (Phase 7b)

Schema proposal (Phase 8)

Composite synthesis (Phase 10)

That's four distinct critical paths dependent on LLM reliability. If DeepSeek has an outage, degrades, or returns malformed JSON, large parts of the platform stop working. Consider:

What's the fallback when LLM is unavailable?

How do you test LLM responses deterministically?

What's the cost exposure at scale?

3. Stagehand/Web Scraping Brittleness
Phase 7b and 8 depend on Stagehand for discovery and fallback. Web scraping is inherently fragile — sites change structure, block bots, or disappear. A single government website redesign could break dozens of discovered sources. The roadmap doesn't show a monitoring/repair mechanism for scraped sources.

4. Human Approval as Bottleneck
Phase 8 requires human approval for every discovered domain. If the agent discovers 50 domains in a week, a human must review 50 schemas. This doesn't scale. The roadmap acknowledges this could become optional "once accuracy is proven," but there's no defined metric for "proven" or automatic rollback if accuracy degrades.

5. PostGIS Dependency for All Spatial Data
The architecture assumes PostGIS for all spatial aggregation. This is fine for UK crime data, but what about:

Global datasets in different coordinate systems?

Very large geometries (country outlines)?

Real-time streaming spatial data?
PostGIS is excellent but may not be optimal for every spatial use case.

DREDGE: Training Videos as Query Visualisation — Value Assessment
Executive Summary
Adding AI-generated training videos as a visualisation type is innovative but high-risk. The concept aligns with DREDGE's mission (unlocking public information) but introduces significant technical, ethical, and operational challenges that the current architecture isn't designed to handle. Recommendation: Prototype as an experimental feature (Phase 14) with strict scope, but do not integrate into core query pipeline until proven.

Strategic Alignment with DREDGE's Mission
What DREDGE Does Well
DREDGE excels at retrieving and visualising factual public data:

Crime statistics (factual, tabular, spatial)

Weather data (factual, temporal, measurable)

Any dataset with defined schema and verifiable source

What Training Videos Would Require
Training videos are procedural, instructional, and often subjective:

"How to get a hunting license" involves steps that vary by state, season, and applicant status

Training requirements change frequently

Regulations differ by jurisdiction

Videos must be accurate enough that someone could follow them and succeed

This is a different category of information — more like "how-to" content than "what-is" data. DREDGE's core value proposition is data intelligence, not procedural guidance.

Value Analysis
Potential Benefits ✅
Benefit	Assessment
User delight	Video responses feel more helpful than raw data. A user asking "how do I get a hunting license" probably wants steps, not statistics.
Differentiation	No other data intelligence platform generates custom instructional videos. Could be a unique selling point.
Long-tail coverage	For queries where no structured data exists, a generated video could still provide value.
Educational use cases	Students, researchers, and citizens could learn about complex processes visually.
Alignment with Phase 10	Semantic queries about "how to" could map to procedural domains.
Significant Risks 🚫
Risk	Severity	Explanation
Accuracy liability	Critical	If a generated video contains wrong steps and someone fails a license exam or breaks a law, who is liable? DREDGE, not the user.
Hallucination in visuals	High	Image generation can create plausible but incorrect documents, maps, or forms. A fake application form could mislead users.
Regulatory variance	High	Hunting licenses vary by state, species, season, and residency. A single video cannot cover all cases.
Staleness	Medium	Regulations change. A video generated today might be wrong next month. DREDGE has no mechanism to expire/refresh videos.
Cost at scale	Medium	Video generation (Remotion + image gen) is computationally expensive. 100 queries = 100 rendered videos.
Storage bloat	Medium	Videos are large. Snapshot model (Phase 8.5) would store videos immutably, multiplying storage costs.
Evaluation difficulty	High	How do you test if a generated video is correct? Unit tests can't watch videos.
User expectations	Medium	If DREDGE starts generating videos, users may expect it for all "how-to" queries, expanding scope indefinitely.
Technical Feasibility Assessment
Required Capabilities vs. Current Architecture
Requirement	Current State	Gap
Query understanding	Phase 10 semantic layer identifies intent	Would need to distinguish "procedural" from "factual" queries
Content generation	None	Would need LLM to generate script, steps, visual descriptions
Image generation	None	Would need integration with DALL-E/Stable Diffusion
Video rendering	None	Remotion.js would need to be integrated and scaled

Adding video generation would require:

New provider type — VideoProvider or RemotionProvider

New storage layer — Object storage (S3) for video files, not Postgres JSON

New delivery mechanism — Streaming endpoints or pre-signed URLs

New cost tracking — Video generation is expensive; needs usage limits

New cache invalidation — Videos become stale; need expiry policy

This is not a small addition — it's a parallel track with its own infrastructure.

Alternative Approaches with Lower Risk
Approach 1: Curated Video Links (Not Generated)
Instead of generating videos, DREDGE could:

Discover existing instructional videos on YouTube, government sites, educational platforms

Store URLs with metadata (source, date, topics)

Return as a "video result" alongside data results

Use Phase 7b/8 discovery to find them

Advantages:

No generation cost

No liability for content accuracy (source is responsible)

Fits existing provider model (URL as source)

Uses existing discovery infrastructure

Disadvantages:

Content may not exist for long-tail queries

Quality varies

Ads, outdated videos, etc.

Approach 2: Structured Step-by-Step (Not Video)
Generate text instructions with optional diagrams:

LLM generates steps from discovered procedural guides

Image generation creates diagrams (maps, forms, checklists)

Return as structured JSON with text + image URLs

Frontend renders as interactive guide

Advantages:

Much smaller storage (text + few images)

Easier to update (regenerate text, not whole video)

Users can scan, print, share steps

Still feels helpful without video production cost

Disadvantages:

Less engaging than video

Still has accuracy liability

Approach 3: Human-in-the-Loop Verified Videos
For high-value procedural domains (licenses, permits, taxes):

Partner with government agencies to host official videos

DREDGE links to authoritative sources only

No generation, just discovery and surfacing

Advantages:

Zero liability

Authoritative content

Builds relationships with data providers

Disadvantages:

Doesn't scale to long tail

Requires partnerships
